# 开发者资源插件推荐网站交互风格决策

## 1. 交互目标
交互风格要服务于一个核心目标：让用户像使用熟悉的开发工具一样，轻松、快速、可预期地找到资源，并能顺手收藏、反馈、提交建议。

优先级排序：
1. **搜索优先**：任何页面都能快速搜索资源。
2. **低学习成本**：不需要教程，打开就知道怎么用。
3. **反馈闭环**：用户发现问题、资源失效、推荐不准时，可以很容易反馈。
4. **轻量可落地**：首版只依赖 GitHub Pages，不引入后端。
5. **操作可信**：收藏、访问官网、提交反馈等操作要有明确反馈。

## 2. 总体交互风格
最终采用：

**命令面板式全局搜索 + 插件市场式资源浏览 + GitHub Issue 式反馈协作。**

这套交互组合适合程序员：
- 命令面板符合开发者使用 Raycast、VS Code、Linear 的习惯。
- 插件市场卡片适合展示大量工具和插件。
- GitHub Issue 反馈方式天然适合静态站，不需要后端，也符合开发者协作习惯。

## 3. 信息架构

### 3.1 顶层导航
首版建议保留 5 个核心入口：
- 首页
- 资源
- 专题
- 榜单
- 收藏

右侧保留两个轻量操作：
- `提交资源`
- `反馈问题`

移动端：
- 顶部保留搜索入口。
- 导航收起为菜单。
- 反馈入口放在菜单底部和页面底部。

### 3.2 页面固定能力
所有页面都应具备：
- 全局搜索入口。
- 返回首页或资源页的清晰路径。
- 反馈问题入口。
- 当前页面状态说明，例如筛选条件、搜索关键词、结果数量。

## 4. 全局搜索

### 4.1 搜索入口
全局搜索是核心交互，必须在所有页面可见或可快速触达。

桌面端：
- 顶部导航中放置搜索框或搜索按钮。
- 支持 `⌘ K` / `Ctrl K` 打开搜索面板。
- 首页首屏使用大搜索框作为视觉中心。

移动端：
- 顶部固定搜索按钮。
- 点击后进入全屏搜索面板。

### 4.2 搜索面板结构
搜索面板采用命令面板风格。

展示内容顺序：
1. 输入框
2. 热门搜索词
3. 最近浏览
4. 收藏资源
5. 搜索结果
6. 无结果建议

搜索结果分组：
- 资源
- 专题
- 分类
- 榜单

### 4.3 搜索范围
搜索字段：
- 资源名称
- 一句话介绍
- 推荐理由
- 标签
- 分类
- 平台
- 适用场景
- 替代方案
- 专题标题

### 4.4 搜索体验规则
- 输入后即时搜索，不需要点击按钮。
- 搜索结果按相关度排序。
- 命中关键词高亮。
- 空结果展示替代建议。
- `Esc` 关闭搜索面板。
- 方向键可移动选中项。
- `Enter` 打开选中结果。

### 4.5 空结果文案
空结果不能只显示“无结果”，要引导下一步。

示例：
- “暂时没有找到相关资源，可以换个关键词，或提交你想推荐的工具。”
- “没有找到 `GraphQL 调试`，试试搜索 `API`、`Postman` 或 `接口调试`。”

## 5. 资源浏览

### 5.1 分类页交互
分类页采用“筛选 + 卡片网格”的插件市场结构。

筛选维度：
- 分类
- 标签
- 平台
- 价格
- 是否开源
- 是否精选

排序方式：
- 推荐优先
- 最新收录
- 热门优先
- 评分最高

交互规则：
- 筛选点击后即时生效。
- 已选筛选条件以标签形式展示，可单独移除。
- 提供“一键清空筛选”。
- URL Hash 中可保留筛选参数，方便分享。

### 5.2 资源卡片交互
资源卡片需要支持快速判断和快速行动。

默认展示：
- 名称
- 简介
- 推荐理由
- 标签
- 价格
- 平台
- 收藏按钮
- 访问官网按钮

悬浮或聚焦时：
- 卡片轻微上移。
- 边框变为主色浅蓝。
- 展示更多操作：查看详情、复制链接、反馈失效。

移动端：
- 不依赖 hover。
- 操作按钮直接可见。

### 5.3 资源详情页交互
详情页采用“先结论、后细节”的结构。

顶部固定信息：
- 资源名称
- 访问官网
- 收藏
- 反馈问题

内容顺序：
1. 推荐结论
2. 适用场景
3. 核心亮点
4. 平台与价格
5. 替代方案
6. 相关专题
7. 反馈入口

## 6. 收藏与最近浏览

### 6.1 收藏规则
首版使用 `localStorage` 存储收藏，不需要登录。

交互规则：
- 点击收藏后立即变更状态。
- 显示轻提示：“已加入收藏”。
- 再次点击取消收藏。
- 收藏页可搜索和筛选本地收藏。

### 6.2 最近浏览
首版可记录最近浏览资源，存储在 `localStorage`。

展示位置：
- 全局搜索面板中。
- 首页下方的“最近看过”模块。
- 收藏页或个人资源库页。

限制：
- 默认保留最近 20 条。
- 提供清空最近浏览入口。

## 7. 问题反馈

### 7.1 反馈入口
反馈入口需要在这些位置出现：
- 顶部导航右侧。
- 资源详情页。
- 资源卡片更多操作中。
- 页脚。
- 空结果状态中。

### 7.2 反馈类型
首版支持 5 类反馈：
- 链接失效
- 信息错误
- 资源不推荐
- 推荐新资源
- 其他建议

### 7.3 首版反馈实现
由于网站只依赖 GitHub Pages，首版不做后端表单。

推荐方案：
1. 主方案：跳转到 GitHub Issue 模板。
2. 备选方案：使用 `mailto:` 打开邮件。
3. 后续方案：接入第三方表单服务，但不作为首版依赖。

### 7.4 GitHub Issue 链接规则
反馈链接可以预填标题和正文。

资源问题标题：
```text
[资源反馈] 资源名称 - 问题类型
```

资源问题正文：
```text
资源名称：
资源链接：
问题类型：
问题描述：
建议修改：
```

推荐新资源标题：
```text
[资源推荐] 工具名称
```

推荐新资源正文：
```text
工具名称：
官网链接：
所属分类：
推荐理由：
适用场景：
价格信息：
替代工具：
```

### 7.5 反馈体验规则
- 反馈按钮不要打断浏览。
- 资源详情页的反馈入口文案建议为：“信息有误？反馈一下”。
- 链接失效反馈应放在“访问官网”附近。
- 提交前明确说明会跳转到 GitHub Issue。

## 8. 提交资源

### 8.1 提交入口
提交资源是增长资源库的关键入口。

位置：
- 顶部导航。
- 首页精选区下方。
- 搜索无结果状态。
- 页脚。

### 8.2 提交流程
首版采用静态流程：
1. 用户点击“提交资源”。
2. 进入提交资源页。
3. 页面说明收录标准和填写格式。
4. 用户点击“去 GitHub Issue 提交”。
5. GitHub 打开预填模板。
6. 维护者审核后更新静态数据。

### 8.3 提交页内容
提交页需要说明：
- 什么资源值得提交。
- 不收录哪些资源。
- 推荐填写哪些字段。
- 提交后如何被处理。
- 链接到 GitHub Issue。

## 9. 综合能力面板

### 9.1 为什么需要综合能力面板
资源网站功能多，但不能让界面变复杂。建议在全局搜索面板中集成一部分综合能力，使用户通过一个入口完成常用操作。

### 9.2 面板支持动作
全局搜索面板除了搜索资源，还可以支持命令：
- `提交资源`
- `反馈问题`
- `查看收藏`
- `查看榜单`
- `查看专题`
- `清空筛选`
- `打开 GitHub 仓库`

### 9.3 命令触发
用户输入 `/` 或关键词时展示命令建议。

示例：
- 输入 `/feedback` 显示“反馈问题”。
- 输入 `/submit` 显示“提交资源”。
- 输入 `/favorites` 显示“查看收藏”。

中文用户也应支持：
- `反馈`
- `提交`
- `收藏`
- `榜单`
- `专题`

## 10. 状态反馈

### 10.1 轻提示
用于短操作反馈：
- 已加入收藏
- 已取消收藏
- 链接已复制
- 筛选已清空

### 10.2 空状态
空状态要给行动建议：
- 没有搜索结果：建议换关键词或提交资源。
- 没有收藏：建议去资源页浏览精选。
- 没有最近浏览：建议从热门分类开始。

### 10.3 错误状态
静态站错误较少，主要是外链或数据问题：
- 外链打开失败无法直接检测，提供反馈入口。
- 数据缺失时显示“信息暂缺”，不要页面崩坏。

## 11. 可访问性
- 所有交互按钮必须可键盘访问。
- 搜索面板打开后焦点自动进入输入框。
- `Esc` 可关闭弹层。
- 弹层关闭后焦点回到触发按钮。
- 颜色不能作为唯一状态表达。
- 标签、按钮、链接要有明确文本。
- 移动端点击区域不小于 44px。

## 12. 首版交互范围

首版必须实现：
- 全局搜索入口。
- 搜索结果即时过滤。
- 分类、标签、平台、价格筛选。
- 排序。
- 本地收藏。
- 最近浏览。
- 反馈问题入口。
- 提交资源入口。
- GitHub Issue 预填链接。
- 空状态和轻提示。

首版暂不实现：
- 登录。
- 云端收藏同步。
- 站内评论。
- 实时投票。
- 后台审核系统。
- 复杂个性化推荐。

## 13. 决策
最终交互风格定义为：

**一个以全局搜索为核心、以资源卡片为主体、以本地收藏和 GitHub Issue 反馈形成闭环的轻量开发者控制台。**

这套交互既符合程序员习惯，也能在 GitHub Pages 纯静态架构下落地。
